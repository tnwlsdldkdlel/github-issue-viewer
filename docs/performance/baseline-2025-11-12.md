## Chrome DevTools Performance 측정 결과

Chrome DevTools Performance API를 사용하여 이미 로드된 페이지의 현재 상태를 측정한 결과입니다. Lighthouse와 달리 "더 보기" 버튼 클릭 후 추가된 이슈도 포함하여 측정합니다.

### 초기 로드 (100개 이슈)

| 항목 | 값 | 비고 |
|------|-----|------|
| DOM 노드 수 | 552개 | - |
| 이슈 아이템 수 | 35개 | PR 필터링 후 실제 이슈 수 |
| 메모리 사용량 | 43.43 MB | JavaScript Heap 사용량 |
| FCP (Performance API) | 0.09초 | 브라우저 Performance API 측정값 |
| DOM Content Loaded | 0.03초 | - |
| Load Complete | 0.31초 | - |

### 대량 데이터 (500개 이슈)

| 항목 | 값 | 초기 로드 대비 | 비고 |
|------|-----|--------------|------|
| DOM 노드 수 | 2,105개 | +1,553개 (+281%) | 3.8배 증가 |
| 이슈 아이템 수 | 139개 | +104개 (+297%) | 4.0배 증가 |
| 메모리 사용량 | 48.48 MB | +5.05 MB (+11.6%) | 약간 증가 |
| FCP (Performance API) | 0.08초 | -0.01초 | 거의 동일 |
| DOM Content Loaded | 0.05초 | +0.02초 | 약간 증가 |
| Load Complete | 0.33초 | +0.02초 | 거의 동일 |

### 성능 차이 분석

**주요 발견 사항:**
1. **DOM 노드 수가 크게 증가**: 552개 → 2,105개 (3.8배)
   - 대량 데이터 렌더링으로 인한 DOM 복잡도 증가
   - 가상 스크롤 미적용으로 모든 이슈가 DOM에 렌더링됨

2. **이슈 아이템 수 증가**: 35개 → 139개 (4.0배)
   - "더 보기" 버튼 4회 클릭으로 약 4배 증가
   - PR 필터링으로 실제 이슈 수는 API 응답보다 적음

3. **메모리 사용량 증가**: 43.43 MB → 48.48 MB (+11.6%)
   - 상대적으로 적은 증가 (React Query 캐싱 효과)
   - 하지만 DOM 노드 증가에 비해 메모리 증가가 적은 것은 좋은 신호

4. **FCP는 거의 동일**: 0.09초 → 0.08초
   - Performance API는 초기 로드 시점만 측정
   - 추가 로드된 이슈는 FCP에 영향 없음

**예상되는 문제점:**
- DOM 노드 수가 2,105개로 많아 스크롤 성능 저하 예상
- 가상 스크롤 미적용으로 인한 렌더링 성능 저하
- 대량 데이터에서 메모리 사용량이 계속 증가할 가능성

### 성능 분석 요약

**Lighthouse 측정 결과:**
- 초기 로드와 500개 이슈 로드 후 측정 결과가 거의 동일함
- Lighthouse는 초기 페이지 로드만 측정하므로 차이가 없음

**Chrome DevTools Performance 측정 결과:**
- DOM 노드 수가 552개 → 2,105개로 3.8배 증가 (대량 데이터 영향 명확히 확인)
- 이슈 아이템 수가 35개 → 139개로 4.0배 증가
- 메모리 사용량이 43.43 MB → 48.48 MB로 약간 증가

**긍정적 측면:**
- CLS가 0으로 레이아웃 시프트 없음
- FCP가 0.76초로 빠른 초기 렌더링
- Speed Index가 1.33초로 양호
- 메모리 사용량 증가가 DOM 노드 증가에 비해 상대적으로 적음

**개선 필요 사항:**
- LCP가 10.66초로 매우 느림 (API 호출 완료까지 대기)
- TBT가 1.0초로 높음 (JavaScript 실행 시간)
- TTI가 10.66초로 매우 느림
- DOM 노드 수가 2,105개로 많아 스크롤 성능 저하 예상

**예상 원인:**
- 500개 이슈를 모두 DOM에 렌더링하여 DOM 노드 수가 많음 (2,105개)
- JavaScript 실행 시간 증가 (React Query, 이슈 목록 렌더링)
- 가상 스크롤 미적용으로 인한 성능 저하
- API 호출 완료까지의 대기 시간 포함

## 측정 방법

### 1. Lighthouse 측정

```bash
# 개발 서버 실행 (다른 터미널)
npm run dev

# 성능 측정
npm run perf:measure
```

결과는 `docs/performance/report.html`에 저장됩니다.

### 2. Lighthouse 자동화 스크립트

Puppeteer를 사용하여 사용자 상호작용을 자동으로 시뮬레이션하고 Lighthouse로 측정합니다.

```bash
# 500개 이슈 로드 후 측정
npm run perf:auto:500

# 다른 시나리오
npm run perf:auto:initial  # 초기 로드
npm run perf:auto:1000      # 1000개 이슈
npm run perf:auto:scroll    # 스크롤 성능
npm run perf:auto           # 모든 시나리오
```

**⚠️ 주의**: Lighthouse는 초기 페이지 로드만 측정하므로, "더 보기" 버튼 클릭 후 추가된 이슈는 측정에 포함되지 않습니다.

### 3. Chrome DevTools Performance 자동화 스크립트 (추천) ⭐

Puppeteer의 Performance API를 사용하여 이미 로드된 페이지의 현재 상태를 측정합니다. Lighthouse와 달리 "더 보기" 버튼 클릭 후 추가된 이슈도 포함하여 측정할 수 있습니다.

```bash
# 500개 이슈 로드 후 측정
npm run perf:devtools:500

# 다른 시나리오
npm run perf:devtools:initial  # 초기 로드
npm run perf:devtools:1000     # 1000개 이슈
npm run perf:devtools:scroll   # 스크롤 성능
npm run perf:devtools          # 모든 시나리오
```

**측정 항목:**
- DOM 노드 수
- 이슈 아이템 수
- 메모리 사용량 (JavaScript Heap)
- Performance API 메트릭 (FCP, DOM Content Loaded, Load Complete)

자세한 내용은 [README.md](./README.md)를 참고하세요.

## 성능 분석

### 현재 확인된 문제점

1. **API 호출 지연**
   - LCP와 TTI가 매우 높음 (10.65초)
   - GitHub API 호출 완료까지의 대기 시간이 포함됨
   - 이는 개발 모드에서 측정했기 때문에 실제 프로덕션에서는 더 빠를 수 있음

2. **JavaScript 실행 시간**
   - TBT가 992ms로 높음
   - React Query와 이슈 목록 렌더링에 시간이 소요됨

3. **메인 스레드 블로킹**
   - Performance 측정에서 긴 작업(Long Tasks) 다수 관찰됨
   - 스크롤 중 메인 스레드 블로킹 발생
   - CPU 스파이크가 빈번하게 발생

4. **예상되는 문제점 (대량 데이터)**
   - 500개 이상 이슈 로드 시 성능 저하 예상
   - DOM 노드 수 증가로 인한 렌더링 지연
   - 스크롤 시 FPS 저하 예상 (가상 스크롤 미적용)
   - 메모리 사용량 증가 예상

### 측정 결과 요약

#### Lighthouse 측정
- **초기 로드 (100개)**: Performance 53점, LCP 10.65초, TBT 1.0초
- **500개 이슈 로드 후**: Performance 53점, LCP 10.66초, TBT 1.0초
- **주요 문제**: LCP 10.65-10.66초, TBT 1.0초 (두 시나리오 모두 동일)
- **긍정적**: FCP 0.76초, CLS 0, Speed Index 1.20-1.33초
- **측정 방법**: 자동화 스크립트 (`npm run perf:auto:initial`, `npm run perf:auto:500`)

**참고**: Lighthouse는 초기 페이지 로드만 측정하므로, "더 보기" 버튼 클릭 후 추가된 이슈는 측정에 포함되지 않습니다. 실제 대량 데이터 성능 차이는 Chrome DevTools Performance로 측정해야 합니다.
