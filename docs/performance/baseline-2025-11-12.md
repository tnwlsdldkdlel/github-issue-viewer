## 측정 방법

### 1. Lighthouse 측정

```bash
# 개발 서버 실행 (다른 터미널)
npm run dev

# 성능 측정
npm run perf:measure
```

결과는 `docs/performance/report.html`에 저장됩니다.

### 2. Chrome DevTools Performance 측정

1. `http://localhost:3003` 접속
2. 개발자 도구 열기 (F12)
3. Performance 탭 선택
4. Record 버튼 클릭 (또는 Cmd+E / Ctrl+E)
5. 페이지 로드 및 스크롤 동작 수행
6. Stop 버튼 클릭
7. 결과 분석
   - 상단 그래프에서 FPS 확인
   - 메인 스레드 시간 확인
   - 긴 작업(Long Tasks) 확인

## 성능 분석

### 현재 확인된 문제점

1. **API 호출 지연**
   - LCP와 TTI가 매우 높음 (10.65초)
   - GitHub API 호출 완료까지의 대기 시간이 포함됨
   - 이는 개발 모드에서 측정했기 때문에 실제 프로덕션에서는 더 빠를 수 있음

2. **JavaScript 실행 시간**
   - TBT가 992ms로 높음
   - React Query와 이슈 목록 렌더링에 시간이 소요됨

3. **메인 스레드 블로킹**
   - Performance 측정에서 긴 작업(Long Tasks) 다수 관찰됨
   - 스크롤 중 메인 스레드 블로킹 발생
   - CPU 스파이크가 빈번하게 발생

4. **예상되는 문제점 (대량 데이터)**
   - 500개 이상 이슈 로드 시 성능 저하 예상
   - DOM 노드 수 증가로 인한 렌더링 지연
   - 스크롤 시 FPS 저하 예상 (가상 스크롤 미적용)
   - 메모리 사용량 증가 예상

### 측정 결과 요약

#### Lighthouse 측정
- **Performance**: 53점 (목표 80점 미달)
- **주요 문제**: LCP 10.65초, TBT 992ms
- **긍정적**: FCP 0.76초, CLS 0

#### Chrome DevTools Performance 측정
- **메인 스레드 시간**: 6,872ms (전체의 12.3%)
- **렌더링 시간**: 883ms
- **주요 문제**: 긴 작업 다수, CPU 스파이크 빈번
- **긍정적**: CLS 0, 메인 스레드 시간 비율 양호

### 개선 방향

1. **우선순위 2 (무한 스크롤)**
   - UX 개선 예상
   - 성능 개선 효과는 제한적일 수 있음
   - 메인 스레드 블로킹 문제는 여전히 존재할 것으로 예상

2. **우선순위 3 (가상 스크롤)**
   - 대량 데이터 렌더링 성능 대폭 개선 예상
   - DOM 노드 수 감소로 인한 성능 향상
   - 스크롤 성능 개선
   - 긴 작업 감소 예상
   - 메인 스레드 블로킹 문제 개선 예상
